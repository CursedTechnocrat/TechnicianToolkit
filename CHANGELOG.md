## [3.6.0] - 2026-06-24

### Added
- **Toolkit-wide version bump to 3.6.** Every tool's `.NOTES Version`, banner v-tag, `Get-TKHtmlFoot -ScriptName` footer, and GRIMOIRE registry `Version` field now reads `3.6`. The GRIMOIRE hub banner now reads `Hub v3.6` and the HEARTH banner reads `HEARTH v3.6`. ANVIL and CLEANSE moved from `3.5.1` to `3.6`. Every tool is again on a single major.minor line (`3.6`), ahead of the first git tag `v3.6.0`.
- **Self-contained portable launcher (`launcher/`) — single-file `.exe` distribution of the whole suite.** A new .NET 8 project (`launcher/TechnicianToolkit.Launcher.csproj`) embeds every root `.ps1`, the shared module `TechnicianToolkit.psm1`, and `config.json` as resources inside one self-contained, single-file executable. At run time `launcher/Program.cs` extracts the embedded scripts to a working directory (next to the `.exe`, or `%TEMP%` if the medium is read-only) and launches `grimoire.ps1` via Windows PowerShell, passing arguments such as `-WhatIf` straight through. Because every script is present locally before GRIMOIRE starts, neither the hub's `Invoke-Tool` nor any tool's module bootstrap ever reaches GitHub — the program runs fully offline with no update checks, which suits a USB-carried field tool on isolated or air-gapped machines. The .NET runtime is bundled, so target machines need only Windows PowerShell (already part of Windows); the existing scripts are embedded verbatim and unmodified. `launcher/build.ps1` produces the `.exe` with one command, and `.github/workflows/release-launcher.yml` builds it for `win-x64` and `win-arm64` on GitHub's runners — attaching both `.exe` files to the GitHub Release on a `v*` tag, or uploading them as workflow artifacts on manual dispatch — so the portable binary can be obtained without a local .NET SDK. The launcher lives entirely under `launcher/`, leaving the root-level Pester compliance checks (which enumerate root `*.ps1` non-recursively) unaffected. A native C# reimplementation of individual tools' logic can later slot into this embed-and-extract structure incrementally, running native when available and falling back to the embedded `.ps1` otherwise.
- **`Show-TKReportResult` — new shared-module helper in `TechnicianToolkit.psm1`.** Standardises the end-of-run experience for report-producing tools: `Show-TKReportResult -Path <file> [-Unattended] [-Label <noun>]` prints a single "report saved" line with the resolved path, then — unless `-Unattended` is set — prompts to open the file in its default handler (`Invoke-Item`). It degrades gracefully when the file is missing (prints a warning instead of prompting) and swallows open failures with a warning. Added to the `Module exports` Pester list.
- **Consistent report endings across the suite.** Before, tools diverged three ways at the end of a run: some prompted "Open report? (Y/N)" (AUSPEX, AUGUR, THRESHOLD, ARTIFACT, CITADEL), some silently auto-opened the browser (PYRE, PALADIN, TOTEM, PORTAL, TALON, BEACON, ANVIL, CODEX, GROVE, CONCLAVE, WRAITH, RITUAL, TENDRIL, TETHER, EXHUME, SCRYER, and others), two (GOLEM, RELIQUARY) auto-opened even under `-Unattended`, and two (WARD, GARGOYLE) saved a report but never offered to open it at all. Every report-producing tool now routes through `Show-TKReportResult`, so each prints the path once and prompts before opening (suppressed under `-Unattended`; for `TALISMAN`/`TENDRIL` the existing `-NoOpen` flag also suppresses it). The GOLEM/RELIQUARY unattended-auto-open bug is fixed as a side effect, and WARD/GARGOYLE gain an open prompt they previously lacked. CODEX, RITUAL, and the two CITADEL reports keep their descriptive labels via `-Label` ("Index", "Rollup report", "Stale accounts report", "Password expiry report"). In `A.R.T.I.F.A.C.T.` the per-run "Report saved" line was lifted out of `Save-HtmlReport` so the helper owns that message at each of its four call sites (one unattended, three interactive cert-menu branches) without double-printing.
- **`G.R.I.M.O.I.R.E.` type-to-find / direct launch.** The hub now resolves free-text input to a tool, so typing `pyre`, `P.Y.R.E.`, `pyre.ps1`, a keyword like `battery`, or a tool number such as `18` at the main prompt jumps straight to the tool instead of requiring a category drill-down. A new `[F]` menu option opens an explicit `Search tools:` prompt, and a tip line advertises the shortcut. Matching (in `Find-Tool`) is: exact key → exact normalised acronym/filename → substring on acronym/filename plus a 3-char-minimum keyword scan of descriptions; acronyms are normalised by stripping dots/spaces/hyphens so `P.Y.R.E.` and `pyre` collapse to one token. Multiple matches render a pick-list (`Select-FromMatches`); a single match launches immediately.

- **Technician notes — new shared-module feature in `TechnicianToolkit.psm1`.** Four exported functions let any tool capture session notes and hand them off to a ticketing system: `Add-TKNote -Text <string> -Category <Info|Action|Warning|Issue|Resolution> [-ScriptName <string>]` records a timestamped entry into a module-scoped buffer; `Get-TKNote` returns the buffer oldest-first; `Clear-TKNote` resets it; `Export-TKNoteReport -Path <file> [-Title] [-ScriptName] [-Ticket] [-Technician] [-Summary]` renders the notes to a dark-themed HTML report built on the shared `Get-TKHtmlHead` / `Get-TKHtmlFoot` template. The report carries four summary cards (total notes / actions / issues / resolutions), a colour-badged notes table (category → `tk-badge-*` tier), and a `Ticket Summary` section containing a plain-text block sized for pasting straight into a ticket comment field. The note buffer lives for one imported-module instance (one tool run) and resets on every `Import-Module -Force`.
- **Pester coverage for technician notes** — new `Describe 'Technician notes'` block exercising buffer add / read / clear, default category, `[ValidateSet]` rejection of an invalid category, insertion order, and `Export-TKNoteReport` (file creation, balanced HTML document, HTML-escape of note text, ticket reference passthrough). The four new functions were also added to the `Module exports` test list.

### Changed
- **`G.R.I.M.O.I.R.E.` banner box-drawing characters fixed.** The ANSI-Shadow "GRIMOIRE" logo had its horizontal rule glyphs corrupted to literal `=` signs (`██╔════╝` rendered as `██╔====╝`, `╚═╝` as `╚=╝`, etc.), so the banner drew broken. Restored the proper Unicode box-drawing characters (`═`) across all six art rows; spacing and width are unchanged (a 1:1 glyph swap).
- **`C.O.N.J.U.R.E.` reworked to front-load operator input and gained an Adobe edition prompt + Google Earth Pro (v3.6).** All human interaction now happens up front in a single PHASE 1 block — package manager, operation (install / upgrade / both), Adobe Acrobat edition, and optional-software selection are collected before any package work begins, so PHASE 2 runs the installs/upgrades start-to-finish with no further prompts (the script no longer stops mid-run to ask about optional software, so it can be kicked off and left alone). New **Adobe Acrobat edition choice**: the operator picks Reader (`Adobe.Acrobat.Reader.64-bit`) or Pro (`Adobe.Acrobat.Pro`) and the chosen package is appended to the required list; Adobe was removed from the static `$RequiredSoftware` / `$RequiredSoftwareChoco` arrays in favour of this prompt. Because Chocolatey's community repo has no Acrobat Pro package, a Pro request under Chocolatey installs `adobereader` and prints a note that, with an Acrobat Pro license, the user can sign in to Reader and run the in-app upgrade to activate Pro. Unattended mode defaults Adobe to Reader. **Google Earth Pro added to the optional list** (winget `Google.EarthPro`, choco `googleearthpro`). New **custom packages** prompt (`Read-CustomPackages`) collected in PHASE 1 right after the optional-software selection: the operator can type their own comma-separated winget/Chocolatey package IDs, which are passed through verbatim to the selected manager and installed as a separate `Custom` batch (its own labelled section in the run output and counted in the installation summary). The package lists were also **consolidated into a single source-of-truth catalog**: `$RequiredCatalog` / `$OptionalCatalog` hold one `PSCustomObject` per app carrying `Name` / `Winget` / `Choco`, replacing the four parallel index-aligned arrays (`$RequiredSoftware` / `$RequiredSoftwareChoco` / `$OptionalSoftware` / `$OptionalSoftwareChoco`) that previously had to be hand-edited in lockstep. A new `Get-PackageId` helper resolves a catalog entry to the ID for the manager chosen at runtime, so winget and Chocolatey IDs can no longer drift out of alignment. Adobe editions moved to `$AdobeReader` / `$AdobePro` catalog entries. README CONJURE section updated.
- **`C.O.V.E.N.A.N.T.` Entra ID join gained an explicit MFA step (v3.6).** The interactive Entra ID branch of the domain-join stage now asks whether the joining account requires MFA, and if so prompts for the method (Microsoft Authenticator push / verification code, SMS code, phone call, hardware / FIDO2 key), explains that a Windows security prompt will appear, and pauses for the technician to confirm readiness before `dsregcmd /join` runs. The method is recorded in the onboarding action log as the `Entra MFA` step. Replaces the previous passive "a browser prompt may appear" note. `-WhatIf` previews the new step; the summary status colouring recognises the `Configured` status.
- **`R.I.T.U.A.L.` now fetches missing tool scripts from GitHub on demand (v3.6).** Previously a recipe step whose tool script wasn't co-located with `ritual.ps1` failed immediately with `Tool file not found` and aborted the recipe — so a lone `ritual.ps1` dropped on a machine (or run standalone from `C:\WINDOWS\system32` after admin elevation) could never complete a recipe, even though the shared-module bootstrap was already self-provisioning `TechnicianToolkit.psm1` from GitHub. A new `Resolve-ToolFile` helper closes that gap: when a step's tool isn't present next to RITUAL it downloads the script from `https://raw.githubusercontent.com/CursedTechnocrat/TechnicianToolkit/main/<tool>`, validates its syntax with the parser (deleting and reporting the file if validation fails) exactly as the module bootstrap does, and only marks the step `Missing` if both the local lookup and the download fail. Co-located tools resolve to their local path unchanged, so installed estates see no behaviour change.

## [3.5.0] - 2026-05-11

### Added
- **Toolkit-wide version bump to 3.5.** Every tool's `.NOTES Version`, banner v-tag, `Get-TKHtmlFoot -ScriptName` footer, and GRIMOIRE registry `Version` field now reads `3.5`. The GRIMOIRE hub banner now reads `Hub v3.5` and the HEARTH banner reads `HEARTH v3.5`. PYRE moved from `3.1` to `3.5`. Every tool is now on a single major.minor line so a future "all-tools-on-vN" assertion can hold without exceptions.
- **Pester `Tier-mapper data tables` Describe block** asserting the verdict-driving reference tables in PALADIN, BEACON, and PORTAL stay correctly populated. The block walks each script via the PowerShell AST, locates the named hashtable assignment / function definition, and re-evaluates just that node in the test scope (avoids dot-sourcing the tools whole, which would launch their main flow). Coverage:
  - **PALADIN** -- `$AsrRuleNames` covers >= 16 well-known ASR GUIDs, the abused-driver and LSASS-credential-theft GUIDs map to recognisable names, and every key is lowercase (so `Get-MpPreference`'s lowercase output looks them up cleanly).
  - **PALADIN** -- `Get-AsrActionLabel` returns the expected label for action codes 0 / 1 / 2 / 6 and falls back to `Unknown (<n>)` for unmapped codes.
  - **BEACON** -- `$AuthStrength` classifies open / shared / WEP as Insecure, WPA1 family as Weak, and WPA2 / WPA3 / OWE as Strong.
  - **BEACON** -- `$CipherStrength` classifies none / WEP as Insecure, TKIP as Weak, AES and GCMP as Strong.
  - **PORTAL** -- `$AuthStrength` classifies Pap as Insecure, Chap as Weak, MSChapv2 as Acceptable, Eap and MachineCertificate as Strong.
  - **PORTAL** -- `$EncryptionStrength` classifies NoEncryption as Insecure, Optional as Weak, Required and Maximum as Strong.
- **`R.I.T.U.A.L.` recipe estate expanded** to use the new endpoint-audit tools.
  - **`HealthCheck` recipe gained PALADIN** as a final step (now seven steps: AUSPEX -> WARD -> THRESHOLD -> AUGUR -> GARGOYLE -> ARTIFACT -> PALADIN). The quarterly machine review covers the Defender posture by default.
  - **New `SecuritySweep` recipe** (five steps: SIGIL -> TALON -> TOTEM -> PALADIN -> ARTIFACT). Read-only security posture rollup that pairs with the existing `TenantSweep`: SecuritySweep covers the endpoint, TenantSweep covers the cloud control plane.
  - **New `NetworkSweep` recipe** (four steps: LEYLINE -> LANTERN -> BEACON -> PORTAL). Read-only network posture rollup combining live diagnostics, LAN discovery, Wi-Fi profile audit, and VPN tunnel inventory.
  - The `[ValidateSet]` on `ritual.ps1`'s `-Recipe` parameter, the header docstring, and the README RITUAL section were updated to surface all six built-in recipes.
- **`CLAUDE.md` Tool Distinctions section gained three new entries** clarifying overlap-prone tool pairs: PALADIN vs SIGIL (audit vs enforce on the Defender surface), BEACON vs LANTERN (saved Wi-Fi config vs current LAN discovery), and PORTAL vs LEYLINE (static VPN configuration audit vs live network troubleshooting).
- **New tool: `P.O.R.T.A.L.`** (`portal.ps1`, Network & Remote key 34) — Profiles, Observes & Reports Tunnels, Authentication & Links. VPN / Always-On VPN audit. Enumerates built-in Windows VPN connections via `Get-VpnConnection` (user scope) and `Get-VpnConnection -AllUserConnection` (all-user / Always-On candidate scope) and surfaces tunnel type (PPTP / L2TP / SSTP / IKEv2 / Automatic), authentication methods (PAP / CHAP / MS-CHAPv2 / EAP / MachineCertificate) classified into Insecure / Weak / Acceptable / Strong tiers, encryption level (NoEncryption / Optional / Required / Maximum) with matching tier badges, split-tunnel flag, connection state, and profile type per row. Always-On app triggers via `Get-VpnConnectionTrigger`, NRPT entries via `Get-DnsClientNrptPolicy`, active VPN tunnel interfaces via `Get-NetIPInterface` (filtered by alias heuristic to avoid swamping the table with IPv6 pseudo-interfaces), and a curated catalog of third-party VPN client signals matched against the local service table: Cisco AnyConnect / Cisco Secure Client (`vpnagent`, `csc_vpnagent`), Palo Alto GlobalProtect (`PanGPS`, `PanGPA`), Ivanti / Pulse (`JuniperNetworksTunnelService`, `PulseService`), OpenVPN (interactive, legacy, GUI), WireGuard (manager and per-config tunnels via wildcard match), Tailscale, ZeroTier, Cloudflare WARP, NordVPN, Proton VPN, F5 BIG-IP Edge Client. Red / yellow / green verdict: red on PAP authentication or NoEncryption (`Write-TKError` telemetry fires for both -- credentials in cleartext or traffic in cleartext are immediate findings); yellow on CHAP, Optional encryption, split-tunnel without a matching NRPT entry for the VPN's DNS suffix, and multiple competing third-party VPN vendors installed concurrently. Special-case `NONE CONFIGURED` posture statement (info-level, not a finding) when zero built-in VPNs and zero third-party clients are detected. Dark-themed HTML report with six summary cards and six detail sections.
- **New tool: `B.E.A.C.O.N.`** (`beacon.ps1`, Network & Remote key 33) — Broadcasts, Encryption, Authentication & Connections Of Networks. Wi-Fi profile audit. Exports each saved WLAN profile via `netsh wlan export profile name="..." folder="..." key=clear` to a per-run temp folder, then parses the locale-stable WLAN profile XML schema (avoids the brittle locale-dependent text output of `netsh wlan show profile`). Surfaces SSID name / hex / hidden flag, connection type, connection mode, autoSwitch, authentication (`open`, `WEP`, `WPA`, `WPAPSK`, `WPA2`, `WPA2PSK`, `WPA3SAE`, `WPA3ENT`, `OWE`, etc.), encryption cipher (`none`, `WEP`, `TKIP`, `AES`/CCMP, `GCMP`), 802.1X usage, MAC randomization, and key material per profile. Adapter inventory via `Get-NetAdapter -Physical` filtered to `Native 802.11`. Filtered tables for open / weak profiles and auto-connecting profiles with risk-tier badges. Red / yellow / green verdict: red on open auto-connect, WEP, deprecated cipher; yellow on TKIP, autoSwitch, hidden+auto, MAC randomization off, > 25 saved profiles. Privacy default is to mask key material in the report; cleartext rendering is opt-in via `-IncludeKey`. Telemetry via `Write-TKError` on per-profile XML export / parse failure. The temp export folder is wiped after parsing in a `try/finally` block. Dark-themed HTML report with six summary cards and five detail sections.
- **New tool: `P.A.L.A.D.I.N.`** (`paladin.ps1`, Security key 26) — Protection Auditor: Logs Antivirus, Defender, Intrusions & Notifications. Antivirus / Microsoft Defender health audit. Reads `Get-MpComputerStatus` for core state (AV / antispyware / AM service, real-time / behavior / IOAV / on-access / NIS toggles, tamper protection, AM running mode, engine and product versions), `Get-MpPreference` for cloud / sample submission posture (`MAPSReporting`, `SubmitSamplesConsent`, `CloudBlockLevel`, `PUAProtection`), exclusions (path / extension / process / IP), and ASR rules (every configured GUID resolved to its friendly Microsoft-published name with mode Block / Audit / Warn / Not Configured). Signature freshness across AV / antispyware / NIS with configurable yellow / red thresholds via `-SignatureMaxAgeDays` (default 7 days yellow, double red). Scan history (last quick / full scan with start, end, age). Threat history via `Get-MpThreat` (every cached threat with severity badge, active vs resolved, detection count, affected resources) and recent detections via `Get-MpThreatDetection` (last 50 with process / user / cleanup-success). Third-party AV products via the `root\SecurityCenter2` namespace with the packed `productState` decoded into real-time and up-to-date flags. Service health for `WinDefend`, `WdNisSvc`, `Sense`, `WdFilter`, `SecurityHealthService`, and `mpssvc`. Recent Defender events from `Microsoft-Windows-Windows Defender/Operational` with configurable lookback via `-EventDays` (default 7, range 1-90). Red / yellow / green verdict drives off Defender disabled, real-time off, tamper off, signature age above the red threshold, unresolved high / severe threats, and critical service failures. `Write-TKError` telemetry on `Get-MpComputerStatus` failure and on each unresolved high / severe-severity threat so an active infection emits a Teams webhook ping when one is configured. Dark-themed HTML report with six summary cards and twelve detail sections.
- **New tool: `C.O.D.E.X.`** (`codex.ps1`, Diagnostics & Reporting key 19) — Compiles Output Documents into an EXhibit. Toolkit report index builder. Walks the configured log directory (or a `-LogDir` override), finds every TechnicianToolkit-generated HTML report whose filename matches `<TOOL>_YYYYMMDD_HHMMSS.html`, groups them by tool prefix, and emits a single dark-themed HTML rollup with relative links to each child report. Variants like `PYRE_battery_report_*` and `CITADEL_StaleAccounts_*` are surfaced as a badge inside the parent tool's section. Optional `-DaysBack <int>` filters to reports younger than N days. Six summary cards (total reports / distinct tools / last-7-days count / total disk size / newest / oldest) and one section per tool with a per-report table. Files outside the timestamp pattern are skipped on purpose so random HTMLs don't pollute the index; CODEX excludes its own outputs from the scan. Distinct from R.I.T.U.A.L. — RITUAL produces a rollup of a fresh recipe run, CODEX answers "what reports already exist on disk?" for ad-hoc work that did not go through a recipe.
- **`P.Y.R.E.` enriched with `powercfg /batteryreport` data (v3.1).** PYRE now runs `powercfg /batteryreport /xml` and parses the XML to surface data the live `ROOT\WMI` classes do not expose: per-battery serial number and manufacture date, the full-charge capacity history (degradation trend over the machine's lifetime), Windows' own runtime estimates for active use and connected standby at both current full charge and original design capacity (so the technician can quote the runtime the user has lost to wear), and aggregated AC-vs-DC time totals across the runtime history. A second `powercfg /batteryreport` run produces the full Microsoft-formatted HTML, saved alongside `PYRE_*.html` and linked from the new `powercfg Battery Report` section. Falls back gracefully with an in-report explanation when powercfg is unavailable (no battery / locked-down policy / older Windows).
